# Data inventory — what's actually in the records

Empirical breakdown from **your real transcripts**: 38 sessions across 6 projects
(DataVine c7-connector + sipi-laravel + worktrees, software boxBot + HAID), 7,057
records, Claude Code versions 2.1.92 → 2.1.156. Generated by
[`research/analyze_records.py`](../research/analyze_records.py) and
[`research/probe_shapes.py`](../research/probe_shapes.py); full dump in
`research/_analysis.json`.

This is the doc that answers "how much can we auto-tag vs. infer." Short version:
**the entire graph skeleton — information-in, actions-out, changes, costs, and the
causal spine — is deterministically extractable. Inference is needed only for
intent labels, ambiguous trigger attribution, and the waste/quality judgment.**

## The spine: ordering & causality

- **Linear order = `timestamp`** within an *agent scope* (`agentId`; main chain has
  none/null, each subagent shares one). This is the "linear order of actions" — the
  reliable backbone.
- **Causal tree = `parentUuid` ← `uuid`.** Mostly linear: only **273 / 6,744
  parents (~4%) have multiple children.** Branching comes from sidechains,
  compaction (`logicalParentUuid` adds an alternate link), and re-edited/retried
  messages — *not* normal flow. So treat the session as a linear spine with a few
  meta-branches, ordered by timestamp.
- **Tool call ↔ result pairing is deterministic.** Result-bearing `user` records
  carry a `tool_result` block whose `tool_use_id` → the `tool_use.id` (verified 100%
  across 7509 results; there is **no** top-level `sourceToolUseID`), plus
  `sourceToolAssistantUUID` (→ the assistant *turn* only). No matching heuristics needed.
- **Sidechain flag:** `isSidechain` true on 1,491 records (subagent work), false on
  5,566.

## Information in & changes — `toolUseResult` is richly structured

`toolUseResult` is a **dict 1,588×**, str 124×, list 19×. The structured payloads
are the single biggest find — they hand us the analysis primitives:

| tool | result keys | what it gives us |
|---|---|---|
| `Read` | `type`, `file{filePath, content, numLines, startLine, totalLines, truncatedByTokenCap}` | exact range read + truncation flag → **redundant-read & unused-context detection, deterministic** |
| `Edit` | `filePath`, `oldString`, `newString`, **`structuredPatch[]`**, **`originalFile`**, **`userModified`**, `replaceAll` | precise diff hunks + full pre-edit content + whether the **user hand-edited** |
| `Write` | `filePath`, `content`, `structuredPatch[]`, `originalFile`, `userModified`, `type` | same, for whole-file writes |
| `Bash`/`PowerShell` | `stdout`, `stderr`, `interrupted`, `isImage`, `noOutputExpected`, `returnCodeInterpretation?`, `backgroundTaskId?`, `persistedOutputPath?` | command outcome (success/fail is semi-deterministic — see gaps) |
| `Grep` | `mode`, `filenames[]`, `numFiles`, `content?`, `numLines?`, `appliedLimit?` | what the search surfaced |
| `Glob` | `filenames[]`, `numFiles`, `truncated`, `durationMs` | discovered files |
| `Agent` | `agentId`, `status`, `totalTokens`, `totalToolUseCount`, `usage`, `toolStats`, `outputFile`, `prompt` | **subagent cost & linkage rolled up** |
| `TodoWrite` | `oldTodos[]`, `newTodos[{content,status,activeForm}]` | task-state deltas |
| `WebSearch`/`WebFetch` | `results`/`result`, `query`/`url`, `durationSeconds` | external info pulled in |

### `structuredPatch` — the goldmine

```
structuredPatch: [ {oldStart, oldLines, newStart, newLines, lines:[...]} , ... ]
```

This is a unified-diff hunk list **provided by Claude Code itself.** Consequences:

- **Re-touched lines is near-trivial:** map each edit's hunk line-ranges to a
  region; ≥2 edits hitting overlapping ranges = rework. No need to reimplement diff.
- **`originalFile`** usually gives the full pre-edit file content → state
  reconstruction without git, and the input to anchor-based region identity. **But it is
  *sometimes* `None`** (observed on large files), so don't assume it's populated — fall back to
  the earliest captured `originalFile` window-wide, then to replaying `structuredPatch` hunks.
- **`userModified`** is a per-edit boolean flagging that the **user changed the file
  out-of-band before this edit** — this *partially closes the "non-tool changes are
  invisible" gap* I'd flagged: tool-channel hand-edits are actually signalled here.

## Cost — `usage` on every assistant record (complete)

All 4,210 assistant records carry: `input_tokens`, `output_tokens`,
`cache_creation_input_tokens`, `cache_read_input_tokens`, `cache_creation` (1h/5m
ephemeral split), `service_tier`, `speed`, `server_tool_use` (web counts),
`inference_geo`, and **`iterations[]`** (per-inference breakdown). Output-token
distribution across the corpus: median 576, p90 3,738, max 32,000.
→ **Token attribution per turn is fully deterministic**, including the
cache_read-resident-context story.

## Subagents — clean, deterministic stitching

- Files: `<session>/subagents/agent-<id>.jsonl` + `.meta.json`.
- **`.meta.json` = `{agentType, description, toolUseId}`** — `toolUseId` links the
  subagent file back to the **exact parent `Agent` tool_use**. (21 have
  agentType/description; 18 have toolUseId.)
- The parent `Agent` result also carries `agentId`, `outputFile`, `totalTokens`,
  `toolStats`. Records inside carry `agentId` + `isSidechain:true`.
→ Parent↔subagent linkage and subagent cost rollup are **100% deterministic.** The
canonical case (subagents writing myopic tests) is fully reconstructable.

## Compaction — present and fully structured ✅

Confirmed in real sessions (both `manual` and `auto` triggers):

- A `system` record, `subtype:"compact_boundary"`, with:
  ```
  compactMetadata: { trigger: "manual"|"auto", preTokens, postTokens,
                     preCompactDiscoveredTools[], durationMs }
  ```
  Real examples: manual 262,707→6,702 tokens; auto 168,373→7,453. Carries
  `logicalParentUuid` (links across the compaction) and `isMeta:true`.
- The summary itself is a `user` record flagged **`isCompactSummary:true`**.
- An `attachment` subtype **`compact_file_reference`** re-injects file context post-
  compact.
→ Compaction boundaries, **magnitude** (pre/post tokens), and the summary are all
deterministically locatable. "Agent acted on an instruction compaction dropped"
becomes a real, detectable comparison (pre-compaction turns are retained on disk).

## Control / attribution / hooks (auto-tags for free)

- **`system` subtypes:** `stop_hook_summary` 84 · `compact_boundary` 2 ·
  `api_error` 1. Hook records carry `hookInfos`, `hookErrors`,
  `preventedContinuation`, `stopReason`, `toolUseID` → **hook activity is auditable**
  (and HAID recommends hooks, so it can later confirm they fired).
- **Attribution fields on assistant records:** `attributionSkill`,
  `attributionAgent`, `attributionPlugin`, `attributionMcpServer`,
  `attributionMcpTool` → each action is auto-tagged with what produced it.
- **Permission mode:** `mode` records (24) + `permissionMode` on user records track
  plan/acceptEdits/auto transitions.
- **`api_error`** system records + assistant `isApiErrorMessage` + system
  `retryAttempt`/`maxRetries`/`retryInMs` → API-level retries are visible (distinct
  from agent retry loops).

## Known gaps / things still to pin down

- **Bash success/failure — RESOLVED (2026-06-06).** The structured `toolUseResult` has
  no exit-code field (only `stdout`/`stderr`/`interrupted`/`noOutputExpected`/occasional
  `returnCodeInterpretation`), so this looked like it needed a Tier-2 heuristic. It
  doesn't: a failed call is flagged by **`is_error: true` on the `tool_result` content
  block** (uniform across all tools incl. Bash; error results carry an `Exit code N`
  prefix and no `toolUseResult` dict). A clean Tier-1 boolean after all — see
  [open-questions V6](../plans/open-questions.md).
- **Large outputs overflow** to `tool-results/*.txt` and Bash to
  `persistedOutputPath`; the inline payload is truncated (`truncatedByTokenCap` on
  reads). Resolve from the sidecar files when needed.
- **`userModified`** covers tool-channel hand-edits but **not** formatter-on-save or
  build-step changes — those are the (~0–1% measured) out-of-band gap, not the scored diff
  (which is replayed from the transcript, not git). (A `sed -i` *issued as a transcribed Bash
  call* IS parsed as a write by `bash_write.py` — detected, content not recovered, flagged;
  **heredoc writes (`cat > f <<EOF`) DO have their content recovered** and feed the diff; only
  shell writes run *outside* the tool stream stay invisible.)
- **Resumed-session duplication** not yet measured here (ccusage dedups by record
  id — replicate). Affects every count if unhandled.
