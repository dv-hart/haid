HAID — visual vocabulary

The channels the bus diagram is built from, in isolation, so we can tune each before wiring it up. Every panel ends with the open decision it teases out. Channels are kept segregated: file identity owns color in the gutters; token-type owns color only in the right-margin ribbon.

Width — token volume (log-tiered)

Width = authoring/context cost, log-mapped to a few discrete gauge tiers on a global fixed scale (same width = same tokens in every view). Discrete reads more reliably than continuous and avoids false precision.

DECIDE: the 6 tier cutoffs (50 / 200 / 1k / 4k / 16k tok) and stroke widths — tune on real sessions. Top "mega" tier gets a token badge so a capped 500k bus stays honest.

Color — file identity

Golden-angle HSL → maximally-distinct, repeatable per file. But a busy session has 30+ files and the palette runs out of perceptual distinctness fast (see neighbours collide).

DECIDE: pure golden-angle vs color-by-directory + per-file shade vs "color only the interesting files, grey the rest." 31 files here; collisions start ~12.

Edges — direction & kind

Left gutter = inbound (reads), right = outbound (writes). A shell write (sed -i / > f) is real but costs ~0 context tokens, so it draws at the min-width clamp, marked (dashed + ~) — present but never fattened.

DECIDE: how to show a file that's both read AND written — both gutters (shown) vs one side.

Waste overlays — the metrics, made visible

The diagram renders the same Tier-1/2 numbers the report ranks. A fat unused bus, a re-touch cluster, a retry loop — each gets a badge keyed to the metric, shown only in overlay mode.

DECIDE: badge vs glow vs hatch fill; show only above-baseline instances (avoid flagging normal iteration).

Right margin — cumulative cost

A secondary scale, not a ruler: gridlines land where the running normalized-token total crossed each threshold, so tick density = a free spend heatmap. Bunched = expensive stretch.

DECIDE (real finding): in this corpus cumulative nTok is dominated by per-turn cache reads (79M for one session). Honest, but it flattens the heatmap. Keep cache-read in the margin tally, or show a second "fresh tokens" axis?

Token-type ribbon — candidate, NOT v1

To spot cost regime shifts ("by step 30 it's all cache reads") — a stacked-area ribbon split by type, Okabe–Ito categorical palette (never ratio→RGB: muddy + colorblind-unsafe). Reserves 4 hues kept out of the file palette.

DECIDE: validate the plain tally first; adopt the ribbon only if regime-shift reading earns the 2nd color language.

Y axis — step vs wall-clock

Step-based (one equal row per call/turn) stays readable where wall-time doesn't — a 3s grep and a 5-min build shouldn't occupy wildly different height, and idle gaps would swamp the canvas.

DECIDE (leaning step): the two margin scales restore time/cost truth; their divergence is itself diagnostic (slow-but-cheap = thinking; fast-but-expensive = a generation burst).
